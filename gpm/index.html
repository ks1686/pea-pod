GPM

Global Package Manager ✦ Track, sync, and reproduce your environment

Get Started

A thin, declarative layer on top of your existing package managers. Define once, install anywhere.

$ go install github.com/ks1686/gpm@latest
⭐ View on GitHub ↓ Download Binary
Features
🌍

Cross-Platform

Works seamlessly on Linux, macOS, and Windows (via WSL2). GPM detects the available package managers on your current host automatically.

📄

Declarative gpm.json

Declare your packages in a single gpm.json file. Commit it, share it, version it — reproduce your environment anywhere.

🔍

Smart Resolution

GPM automatically finds the intersection of available managers and the ones that carry your package, scoring by your preferred manager and built-in priority.

🔒

Version Pinning

Pin exact versions with semver constraints like 0.10.*, or use * for always-latest. Reproducibility is first-class.

👁️

Dry-Run First

Preview exactly what commands will run before committing. gpm apply --dry-run builds a full plan and shows it to you.

⚡

Minimal Learning Curve

Inspired by NixOS's declarative model — without the complexity. If you know how to use a package manager, you already know GPM.

Quick Start 
# Add packages — each one is tracked in gpm.json and installed immediately
gpm add git
gpm add neovim --version "0.10.*"
gpm add firefox --manager flatpak:org.mozilla.firefox

# See what is currently installed by gpm (reads gpm.lock.json)
gpm list

# Edit gpm.json directly in your $EDITOR
gpm edit

# Reconcile — installs newly added packages, removes deleted ones
gpm apply --dry-run   # preview the delta first
gpm apply             # apply it

# Remove a package — uninstalls it and removes it from the spec
gpm remove git
CLI Reference 
gpm add <id> [--version <ver>] [--prefer <mgr>] [--manager <mgr:name,...>]
gpm remove <id>
gpm list
gpm apply [--dry-run] [--strict]
gpm edit
gpm help
  • --version <ver> — version constraint, e.g. "0.10.*" (for gpm add)
  • --prefer <mgr> — preferred manager, e.g. brew (for gpm add)
  • --manager <mgr:name,...> — manager-specific names, e.g. flatpak:org.mozilla.firefox (for gpm add)
  • --dry-run — print the reconcile plan without executing (for gpm apply)
  • --strict — exit with an error if any package cannot be resolved (for gpm apply)
  • --file <path> — path to gpm.json, default: ./gpm.json (all commands)
gpm.json Format 
{
  "schemaVersion": "1",
  "packages": [
    {
      "id": "git"
    },
    {
      "id": "neovim",
      "version": "0.10.*",
      "prefer": "brew"
    },
    {
      "id": "firefox",
      "managers": {
        "flatpak": "org.mozilla.firefox",
        "brew":    "firefox",
        "snap":    "firefox"
      }
    }
  ]
}
  • id — canonical name for the package (used by gpm)
  • version — optional version constraint; omit for latest
  • prefer — optional hint for which manager to use first
  • managers — optional map of manager-specific identifiers for packages with different names across managers
How Resolution Works
1️⃣

Detect

GPM detects which package managers are available on the current host.

2️⃣

Honour Preference

Honours the prefer hint if that manager is available on the host.

3️⃣

Managers Map

Falls back to the first available manager listed in the managers map.

4️⃣

Registry Fallback

Falls back to the first available manager in the registry, using the package ID as the name. Unresolved packages produce a warning; use --strict to treat them as errors.
